| 2026-05-26 (chat-stop redesign) | **Per-session task cancellation, not a hard halt.** Initial chat-side `stop` implementation routed all stop variants through the existing `data/STOP` sentinel, which halts the autonomous mind + scheduler + every in-flight `agent.run()` across every session — catastrophic blast radius for what an operator means when they type "stop" mid-conversation. Operator caught it immediately during the live test; redesigned the same day. **New default**: bare `stop` / `halt` / `kill` / `pause` (or any natural-language equivalent the LLM understands) cancels JUST the in-flight `run_session` task for the current chat session. Autonomous mind keeps running. Scheduler keeps firing. Other sessions unaffected. No sentinel write. **Hard kill stays opt-in**: `stop --hard` / `stop --all` / `stop --cancel-goals` / `stop --cancel-schedules` writes the sentinel (and optional DB cancels) — explicit destructive flag required. **Implementation**: `core/gateway.py` tracks `_inflight_run_tasks: dict[session_id, asyncio.Task]`; `_handle_chat` wraps `run_session` as a task, stores it, catches `CancelledError`.
